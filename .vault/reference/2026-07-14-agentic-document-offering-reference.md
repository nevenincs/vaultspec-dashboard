# `agentic-document-offering` reference: `existing workspace, feedback, and authoring seams`

Dashboard snapshot: `f69bbcdbca2f3bb640627580c69c68d3af4ea5b1`

A2A snapshot: `ad9d5e3102bedc3bf9717015c246bd1558249ba3`, plus the active
uncommitted ADR-authoring orchestration work described below.

This reference audits the frontend shell and stage, document viewer, comments plane,
authoring client, Rust authoring API, and Figma binding that constrain the agentic
document offering.

## Summary

The current product has three distinct planes: a central Dockview workspace containing
documents and the graph placeholder, durable comments attached to Markdown heading
sections, and a governance Review Station for ledgered changes. The missing surface is
an explicit central `browse | create` workspace mode with the Codex/Claude two-tier
composer, deliberate A2A team and model-profile launch context, generated content in the
existing document grammar, and structured feedback in later agent turns.

### Central workspace seam

`AppShell` mounts one central stage row containing `DockWorkspace`
(`frontend/src/app/AppShell.tsx:274`). `DockWorkspace` recognizes `graph` and `doc`
components (`frontend/src/app/stage/DockWorkspace.tsx:63-65`), reconciles graph
visibility at `frontend/src/app/stage/DockWorkspace.tsx:349`, and reconciles document
tabs at `frontend/src/app/stage/DockWorkspace.tsx:384`. `GraphCanvasHost` and the
Dockview container are siblings (`frontend/src/app/stage/DockWorkspace.tsx:437-439`).

The safe amendment is an explicit view-local `workspaceMode: "browse" | "create"` and
one structural create panel in the Dockview registry. The create panel must be excluded
from document close, ordering, and tab synchronization. Browse document panels should
reconcile out while Create is active so their observers unmount, while the open-document
store and browse layout remain intact for restoration.

The existing vault or code `BrowserMode` and the persistent graph visibility preference
must not be overloaded. They represent different user intent.

### Portal-pinned graph constraint

`GraphCanvasHost` owns the app-lifetime stage
(`frontend/src/app/stage/GraphCanvasHost.tsx:23-71`). `canvasPin` states that the canvas
is never re-parented and exposes visibility and geometry synchronization
(`frontend/src/app/stage/canvasPin.ts:1`, `:73`, `:97`, `:166`). `GraphPanel` is only
the layout placeholder and timeline tether (`frontend/src/app/stage/GraphPanel.tsx:32-77`).

Branching in `AppShell` between `DockWorkspace` and an agent surface would unmount the
canvas host. Re-parenting it would violate the graph contract. Covering the browse
workspace with an overlay would keep hidden document observers active. The create
workspace must therefore live inside Dockview while the canvas host remains its
app-lifetime sibling.

### Documents generated by an agent

The tab store is the source of truth for open documents while Dockview owns geometry.
`deriveDockWorkspaceSyncPlan` lives at `frontend/src/stores/view/tabs.ts:117`;
`useDockDocPanelView` at `:330`; `openDocTab` at `:391`; and the aggregate workspace view
at `:656`. `DocPanel` routes canonical Markdown and code content
(`frontend/src/app/stage/DocPanel.tsx:33`). `MarkdownDocView` begins at
`frontend/src/app/viewer/MarkdownDocView.tsx:136`.

Existing tab restoration, Dockview lifecycle, and Markdown reader behavior are reusable.
A generated document cannot use the canonical worktree document endpoint before it is
applied. It needs a proposal or run-output source with exact revision identity, plus a
structural panel id that document synchronization ignores. This is a data-source
difference, not a reason to invent a separate visual artifact system.

### Manual creation remains separate

`CreateDocDialog` begins at `frontend/src/app/left/CreateDocDialog.tsx:124`. Submission
starts at `:237`; the feature and document pickers use `AutocompleteCombobox` at `:498`
and `:785`. Its mutation sends `operation: "create_document"` through the ledgered
direct-write client (`frontend/src/stores/server/queries/mutations.ts:778-795`). It
creates a scaffold immediately and does not invoke an agent or generate a body.

Feature selection, feature coverage, eligibility explanations, title validation, and
opening the resulting canonical document are reusable. The modal should remain the
secondary manual path, not become the agent composer.

### Current A2A thread-launch contract

`CreateThreadRequest` accepts `initial_message`, optional `team_preset`, optional
metadata, and no model or provider field
(`Y:/code/vaultspec-a2a-worktrees/main/src/vaultspec_a2a/api/schemas/rest.py:37-51`).
`POST /api/threads` maps those fields into `ThreadCreationRequest` and dispatches through
the control service
(`Y:/code/vaultspec-a2a-worktrees/main/src/vaultspec_a2a/api/routes/threads.py:53-100`).
`ThreadMetadata` requires an absolute `workspace_root` and optionally carries
`feature_tag`
(`Y:/code/vaultspec-a2a-worktrees/main/src/vaultspec_a2a/context/metadata.py:48-83`).

No preset is not an executable default. `requires_dispatch` returns false for an absent
or empty preset
(`Y:/code/vaultspec-a2a-worktrees/main/src/vaultspec_a2a/thread/creation.py:33-39`),
and the service persists the thread without dispatch
(`Y:/code/vaultspec-a2a-worktrees/main/src/vaultspec_a2a/control/thread_service.py:396-404`).
With a preset, dispatch carries the preset, workspace, initial message, feature, and
metadata to the worker (`thread_service.py:406-448`). The proposed Create surface must
therefore require an eligible team rather than implying that prompt text alone starts
an agent.

The accepted dashboard A2A ADR describes a future five-verb engine gateway. Repository
search finds no implemented `/ops/a2a` route or frontend A2A client in this snapshot.
The browser must continue to talk only to the dashboard engine, so an engine adapter is
required before the live sibling endpoint can power the composer.

### Team discovery and the ADR Research execution target

`GET /api/teams` discovers every preset, loads it, and returns id, display name,
description, topology, and worker count without eligibility or recommendation
(`Y:/code/vaultspec-a2a-worktrees/main/src/vaultspec_a2a/api/routes/teams.py:55-76`,
`Y:/code/vaultspec-a2a-worktrees/main/src/vaultspec_a2a/api/schemas/rest.py:196-209`).
Discovery includes bundled mock and test presets. Rendering this response directly
would expose implementation topology and non-product choices.

The intended offering is **ADR Research**, backed by preset id
`vaultspec-adr-research`. Its preset declares `research_adr`, four workers, and a
Research-to-ADR phase machine
(`Y:/code/vaultspec-a2a-worktrees/main/src/vaultspec_a2a/team/presets/teams/vaultspec-adr-research.toml:1-23`,
`:43-94`). The roles and effective defaults are:

- researcher: Claude, mid;
- synthesist: Claude, high;
- ADR author: Claude, high;
- document reviewer: Zhipu, mid.

Those assignments are defined in the four corresponding agent preset files under
`src/vaultspec_a2a/team/presets/agents/`. They prove that the user control represents a
team-wide model profile, not one model.

This preset is not currently a finished served capability. The committed
`TopologyType` enum contains only star, pipeline, and pipeline-loop
(`Y:/code/vaultspec-a2a-worktrees/main/src/vaultspec_a2a/team/team_config.py:66-75`).
An existing conditional test documents that `RESEARCH_ADR` is pending
(`Y:/code/vaultspec-a2a-worktrees/main/src/vaultspec_a2a/team/tests/test_team_config.py:882-904`).
The active uncommitted `phase_gate.py` adds a `DocumentProposalSubmitter` protocol, but
repository search finds no concrete proposal-submitter wiring into a compiled
`research_adr` graph. The UI may show **ADR Research** as the proposed target while
serving an unavailable reason; it must not claim the run can start.

### Model-profile contract gap

Neither `CreateThreadRequest` nor `DispatchRequest` accepts a request-level model or
provider override. `GET /api/teams` exposes neither effective per-role assignment nor
readiness, and there is no model/profile discovery endpoint. A UI-only model selection
would therefore be decorative and unsafe.

The missing cross-repository contract is a model-profile catalog and launch reference.
The catalog must expose a stable profile id, human label, effective provider and
capability per role for the selected team, availability, eligibility, and unavailable
reason. The initial explicit selection is **Team defaults** with id `team-defaults`.
Send remains disabled until both the selected team and profile are positively eligible.

### Comment anchors and generated targets

Backend comments are durable section-anchored state
(`engine/crates/vaultspec-api/src/authoring/comments.rs:1`). The implementation explicitly
limits anchors to heading sections rather than spans (`:29`). The frontend selector
contains heading path, expected section hash, and optional advisory `range_hint`
(`frontend/src/stores/server/authoringComments.ts:42`). Backend selection and exact
resolution are in `engine/crates/vaultspec-api/src/authoring/sections.rs:35-113`; the
drift test at `:384` confirms that `range_hint` is not identity-bearing.

`sectionAnchor` builds selectors from heading blocks
(`frontend/src/app/viewer/sectionAnchor.ts:127-221`), and `remarkBlockId` stamps only
headings (`frontend/src/app/viewer/remarkBlockId.ts:85`). Current comments support
neither source-line nor selected-text anchors.

A selection-capable amendment needs a tagged selector union: existing heading section;
exact selected quote with bounded prefix or suffix context and content revision or hash;
optional revision-fenced byte offsets; and line numbers for display only. A DOM `Range`
is not stable or authenticated source identity.

Current comment targets contain only canonical `document_node_id`; they cannot address
an unapplied generated artifact. The target must become a union of canonical document
revision or generated session/run, proposal/changeset child, and exact proposal
revision. Existing comment CRUD, attribution, resolution, re-open, delete, re-anchor,
SSE, and orphan handling remain reusable.

### Feedback batches and prompt turns

`StartPromptTurnRequest` carries only `prompt` and optional `summary`
(`engine/crates/vaultspec-api/src/authoring/api/mod.rs:392`). `PromptTurnRecord` preserves
the prompt digest, byte count, actor, and LangGraph references
(`engine/crates/vaultspec-api/src/authoring/session/types.rs:116`). Session and run routes
are registered at `engine/crates/vaultspec-api/src/authoring/http/mod.rs:487-491`.

Flattening unresolved comments into prompt text would discard anchor, actor, target
revision, and comment identity. The durable handoff should be a bounded
`PromptFeedbackBatch`: the server resolves and authorizes selected comment ids, freezes
their bodies and anchor evidence against an artifact revision, links the snapshot to the
new prompt turn, and records consumption without resolving the original comments.

The frontend authoring client exposes proposal, direct-write, review, apply, rollback,
and comment methods but no session, turn, run, or A2A control methods
(`frontend/src/stores/server/authoring/index.ts:267-908`). New store clients, query keys,
mutations, run recovery, and activity handling are required. The live A2A
`POST /api/threads` request has no `feedback_batch_id`; the accepted future edge is also
unimplemented. Feedback continuation is a cross-repository contract amendment.

### Iteration versus governance

Product comments are accumulated content-anchored feedback and do not approve, reject,
or apply anything. Governance operates on proposals and changesets and controls whether
generated work becomes canonical. `ReviewStationSection` begins at
`frontend/src/app/authoring/ReviewStation.tsx:562`; `DiffPanel` begins at
`frontend/src/app/authoring/DiffPanel.tsx:144` and remains useful as a secondary changes
view, not a comment-capable document reader.

The backend lifecycle is:

`prompt -> research and ADR draft -> document comments -> feedback turn -> revised draft -> governance gate -> apply`

The Review Station remains the final authority plane. The standard conversation and
adjacent document become the iteration plane.

### Figma and component inventory

The binding rule requires missing primitives to be added centrally and Figma component
names to match React exports. Reusable kit components include `SearchField`,
`SegmentedToggle`, `Card`, `StateBlock`, `Button`, `IconButton`, and `Segment`.
`AutocompleteCombobox` is reusable for feature selection but is not currently in the
kit.

There is no shared `TextArea`, `PromptComposer`, `ChatInput`, or `MessageInput` React
component. Existing comment entry uses ad hoc textareas. Design-system discovery in the
bound Figma file found no published prompt-composer, chat-input, textarea, message-input,
input-field, or segmented-toggle record. `SearchField` must not be repurposed because
search and agent instruction entry have different semantics and states.

This absence does not authorize a new composer design or a stripped-down text box. The
binding benchmark is the Codex/Claude rounded two-tier composer: prompt above;
auxiliary, attachment, or context controls lower-left; execution and model-profile
controls plus primary action lower-right; project or workspace context adjacent or
immediately below. The baseline text is **Create a new architecture decision…** with no
explanatory hero or workflow copy.

Figma may represent proposed team and model-profile controls before the backend is
implemented. Those frames must show send disabled or the controls unavailable until
served eligibility is positive. Reuse of a maintained primitive remains preferable,
but no dependency is selected by this reference; any implementation must preserve the
benchmark anatomy, accessibility, engine-only browser transport, and VaultSpec tokens.
