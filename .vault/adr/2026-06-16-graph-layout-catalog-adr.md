# `graph-layout-catalog` adr: `hierarchical, radial, and community layouts as framework-free deterministic-seed modes` | (**status:** `deprecated`)

## Context

The dashboard node-graph (`frontend/src/scene/`, PixiJS) ships three representation modes — `connectivity` (the d3-force solver owns positions), `lineage` (a deterministic derivation-DAG seed), and `semantic` (UMAP, v1-gated and held) — plus a separate force/circular render-tuning toggle. The representation framework was settled in `[[2026-06-14-graph-representation-adr]]`: a **mode** is a CPU spatialization of whatever nodes the **lens** serves (the two axes are orthogonal and freely combinable), and a mode is dispatched as either a *deterministic seed* (a populated `positions` Map the solver is held stopped over, the `lineage`/`circular` pattern) or a *solver* (`positions: null`, connectivity). `[[2026-06-15-dashboard-node-graph-stability-adr]]` then retired `graphology`/`graphology-layout-forceatlas2` for `d3-force`, and its D10 scope-fence reaffirmed that lineage and semantic remain deterministic seed layouts with the solver held stopped.

This ADR extends the catalog with the three standard data-science network-visualization layouts the research (`[[2026-06-16-graph-layout-catalog-research]]`) analyzed: **hierarchical/layered (Sugiyama)**, **radial/tree**, and **community-clustered (Louvain)**. All three are deterministic CPU spatializations that plug into the existing seam exactly like `lineage` — one `RepresentationMode` union member, one `case` in `representationLayout.ts:56`, one framework-free `*Layout.ts` module, one chrome segment. The assembly's stop-init-no-start path (`fieldAssembly.ts:615-630`) is already generic over "any mode that returns a positions Map," and the re-seed-on-new-slice path (`fieldAssembly.ts:777-826`) already admits any non-connectivity mode.

The decisions this ADR must rule on are: (a) for each layout, build hand-rolled vs adopt a library; (b) how Sugiyama relates to the existing `lineage` mode; (c) the radial root-selection policy; (d) the community placement idiom (deterministic seed vs grouped-force) and whether to re-adopt the deliberately-retired `graphology`; (e) which exponential strategies the bounded rule forbids; (f) edge-input policy per layout; (g) chrome for a 6+-mode catalog and whether any new mode is gate-downgraded like `semantic`; and (h) removing the dead `sigma` package.json entry.

The governing constraints are the four CLAUDE.md rules: `graph-compute-is-cpu-gpu-is-render-and-search` (layout is CPU, GPU only draws), `graph-queries-are-bounded-by-default` (layouts run over the LOD-bounded slice — hundreds of nodes), `published-wheel-purity` (Python-only; npm bundle weight is the analog concern), and `dashboard-layer-ownership` (all compute in `frontend/src/scene/field/`, chrome only writes the mode to the view store, nothing fetches the engine or reads `tiers`).

## Decision

**D1 — All three new modes are deterministic-seed modes; none is a solver and none touches the engine or the wire.** Each returns a populated `positions: Map<string,{x,y}>` from `representationLayout.ts`; the assembly stops the solver, calls `layout.init(nodeIds, [], seeds)`, holds it stopped, and fits-once — the worked `circular`/`lineage` pattern at `fieldAssembly.ts:546-556` and `:615-630`. No new `SceneController` command, no new wire field, no engine change. *Rationale:* the representation ADR fixed seed-vs-solver dispatch and the seam is already generic over seed modes; adding modes is inheritance, not re-architecture, and keeps every new layout inside the scene layer per `dashboard-layer-ownership`. *Verdict: accept.*

**D2 — Hierarchical (Sugiyama) is hand-rolled, extending `lineageLayout`'s longest-path layering; d3-dag is declined for v1.** The new `hierarchicalLayout.ts` reuses the existing longest-path depth computation (`lineageLayout.ts:82-105`, already memoized and cycle-safe via a visiting guard) for layer assignment, then adds an iterated **barycenter/median crossing-reduction sweep** to order nodes within each layer and a simple intra-layer x-coordinate assignment. Dummy nodes are inserted on multi-layer-spanning edges. `d3-dag` (v1.2.1, MIT) is declined for v1: its only material advantage over a hand-roll is the *optimal* layering/decross/coord strategies, which D6 forbids at scale anyway, and even its heuristic strategies bundle `javascript-lp-solver` + `quadprog` unless reliably tree-shaken. *Rationale:* the research shows the heuristic Sugiyama pipeline is O(V+E) per pass plus O(layer-width²) per sweep — sub-to-low-millisecond at the bounded ceiling — and generalizes code the project already owns; zero new dependency honors the wheel-purity/bundle-weight discipline. d3-dag stays the documented fallback if optimal crossing minimization is ever wanted, restricted to heuristic strategies. *Verdict: accept (hand-roll); d3-dag deferred-as-fallback.*

**D3 — Hierarchical is a DISTINCT mode from lineage, not a replacement or a "richer lineage."** `lineage` stays a single-axis, derivation-edge-only, holding-lane-aware provenance spine (its `onSpine`/`dangling` honest-degradation semantics from `lineageLayout.ts:24-42` are load-bearing for the review task and must not be diluted). `hierarchical` is a general 2-D layered layout over the structural backbone (D7), with full crossing reduction, for *any* DAG-shaped subgraph — not just the research→adr→…→rule provenance chain. *Rationale:* the two answer different questions (lineage = "trace this decision's provenance"; hierarchical = "show this subgraph's layered flow") and the lineage mode's PROV-convention honesty is a product commitment from the representation ADR that a general Sugiyama would erase. *Verdict: accept (distinct mode).*

**D4 — Radial adopts `d3-hierarchy` (ISC, zero runtime deps).** New `radialLayout.ts`: pick a root → derive a BFS spanning tree from that root → `d3.hierarchy` → `d3.tree().size([2π, R])` → polar-to-cartesian → positions Map. *Rationale:* `d3-hierarchy` is ISC-licensed, zero-dependency, no-DOM, and sits exactly beside the already-shipped `d3-force`/`d3-ease`/`d3-interpolate` family — the lowest-risk, highest-reuse add in the catalog. Hand-rolling Reingold-Tilford tidy-tree x-assignment is the one place where a battle-tested zero-dep library clearly beats a hand-roll. *Verdict: accept.*

**D5 — Radial root selection is salience-max, with a documented selected-node override and per-component sectors.** The default root is the highest-`salience` node in the served slice (`SceneNodeData.salience`); degree-max is the tie-breaker; when the user has a node selected, the selected node becomes the root (so radial reads as "hops from what I'm looking at"). The BFS spanning tree gives shortest-path-from-root, so radial distance reads as hops. Disconnected components each get their own per-component root (highest local salience) laid out in **separate angular sectors** of one shared field, not separate concentric rings. *Rationale:* salience is the representation ADR's a-priori-importance / DOI term, so a salience root makes "most important thing at the center" the default; selected-node override serves the focus+context loop; sectors keep components co-visible without implying false ring-distance between them. Determinism: ties broken by salience-then-id sort, mirroring lineage's `sort()`. *Verdict: accept.*

**D6 — The exponential strategies are forbidden.** Neither the hand-rolled Sugiyama nor any future d3-dag fallback may use ILP-optimal crossing minimization (`decrossOpt`) or quadratic-program coordinate assignment (`coordSimplex`/`coordQuad`-optimal). Only the near-linear heuristics ship: longest-path layering, iterated barycenter/median decross, and greedy/Brandes-Köpf-style x-assignment. *Rationale:* `graph-queries-are-bounded-by-default` makes a single super-linear layout a denial-of-service on the whole dashboard; the bounded slice (hundreds of LOD nodes) is exactly the regime where heuristics are within budget and optima are not. This is the layout-time analog of the wire-bounding rule. *Verdict: accept (hard forbid).*

**D7 — Edge input per layout is explicit; all three layouts run on `splitBackbone(edges).backbone`.** Hierarchical and community build their structure from the declared+structural layout backbone (`backbone.ts:41`, `LAYOUT_BACKBONE_TIERS`), the same high-precision subset the connectivity solver is fed and the anti-hairball discipline the representation ADR mandates; temporal/semantic edges remain disparity-thinned layered *context*, never layout input. Radial likewise extracts its BFS tree over the backbone adjacency. The one exception is `lineage`, which keeps its derivation-edge-only input (`isLineageEdge`) — provenance is a different edge semantics than structural flow. *Rationale:* reusing `splitBackbone` keeps every layout honoring the one anti-hairball backbone, avoids each mode re-deriving an edge policy, and matches the connectivity feed at `fieldAssembly.ts:637`. For Sugiyama, cycles in the backbone are removed by the same visiting-guard back-edge break already in `lineageLayout.ts:88`. *Verdict: accept.*

**D8 — Community is a hand-rolled framework-free Louvain; `graphology` is NOT re-adopted.** New `communityLayout.ts`: build adjacency from `splitBackbone(edges).backbone` → a self-contained Louvain (modularity-gain move loop + community-aggregation recursion, seeded RNG) → community labels → deterministic two-level clustered placement. `graphology` + `graphology-communities-louvain` are declined; re-adopting `graphology` purely for Louvain would re-introduce the exact dependency `[[2026-06-15-dashboard-node-graph-stability-adr]]` (D1) deliberately retired. *Rationale:* the research scopes hand-rolled Louvain at ~150 framework-free lines, near-linear in edge count, negligible at the bounded ceiling; honoring the stability cycle's retirement is a standing project commitment and re-adoption for one algorithm is a poor trade. Leiden is deferred (no maintained pure-JS graphology-free implementation). *Verdict: accept (hand-roll); graphology re-adoption declined.*

**D9 — Community placement is a DETERMINISTIC two-level seed, not a grouped-force.** Communities are arranged on a coarse outer circle; each community's members are packed locally (deterministic inner circle-pack/circular arrange, reusing the `circularArrange` idiom). The "grouped force" alternative (feeding community membership as an attraction force into the live d3-force solver) is declined because it re-introduces the solver and breaks the deterministic-seed contract — mode-switch object constancy and mental-map stability both depend on the seed being a pure function of the slice. Louvain's RNG is seeded (deterministic id-sort tie-breaking, mirroring `lineageLayout.ts:131`) so the same slice always yields the same communities and the same placement. Community membership may optionally drive the existing `featureHulls` overlay for cluster hulls — an overlay, never a re-layout. *Rationale:* determinism is the seam contract for seed modes (`fieldAssembly.ts:615`), and a non-deterministic community layout would flicker communities on every re-seed, exactly the mental-map instability the stability ADR fought. *Verdict: accept.*

**D10 — No new mode is gate-downgraded like `semantic`; all three ship un-gated.** The `semantic`-mode gate (`SEMANTIC_MODE_GATE`, `representationLayout.ts:69`) exists because UMAP's promotion was conditioned on a *measured* runtime/legibility trigger and on a wire amendment (per-node embedding vectors). Hierarchical, radial, and community need no new wire data, are near-linear at the bounded ceiling by construction (D6), and use only framework-free or zero-dep CPU code — there is no measurable trigger to gate on. They are available the moment they ship. *Rationale:* the gated-downgrade precedent is specifically for modes blocked on an unmet measurable condition; applying it to modes with no such condition would be ceremony, not honesty. *Verdict: accept.*

**D11 — The chrome moves from a flat segment row to a grouped picker once the catalog reaches 6+ modes.** The current `LayoutGroup` `Segmented` control (`GraphControls.tsx:280-306`, Network/Tree/Grouped/Timeline, where Timeline is the temporal seam, not a spatial mode) does not scale to six spatial modes plus Timeline. The three new modes land as entries in a grouped picker: a "Spatial" group (Network, Tree=lineage, Hierarchical, Radial, Grouped=community, and Grouped-by-meaning=semantic when its gate passes) with Timeline kept as the distinct temporal entry. Labels stay human ("Layered", "Radial", "Communities") consistent with the existing Network/Tree/Grouped naming. The `available` flag the segmented control already carries for held `semantic` (`GraphControls.tsx:299`) is reused for any future gated entry. *Rationale:* a flat seven-wide row is illegible; grouping preserves the orthogonal lens/mode framing and keeps Timeline visibly distinct from the spatial modes, as the current code already flags it. *Verdict: accept.*

**D12 — Remove the dead `sigma` package.json entry.** `sigma` (`^3.0.3`, `package.json:38`) is imported nowhere in `src` — it is the abandoned render half of the retired `graphology`/ForceAtlas2 stack. Drop it. *Rationale:* it is dead weight that misleads a future agent into thinking a sigma render path exists; removing it is the bundle-hygiene completion of the stability ADR's `graphology` retirement, with zero call-site impact. *Verdict: accept (remove).*

## Consequences

**Gains.** The catalog grows from three to six spatial modes with one new dependency (`d3-hierarchy`, ISC zero-dep) and zero re-adopted retired libraries. Each mode is a pure deterministic function of the slice, so mode-switch object constancy and mental-map stability hold by construction (the stability ADR's hard-won property). All compute stays CPU/framework-free in `frontend/src/scene/field/`, so the layer-ownership and CPU-only rules hold with no engine or wire change. The hierarchical mode reuses the longest-path code the project already owns; the community mode reuses the `circularArrange` and `featureHulls` idioms.

**Costs and difficulties.** Three new layout modules plus a barycenter crossing-reduction sweep and a hand-rolled Louvain are real algorithmic code to write, bound, and test against the node ceiling. The chrome refactor from a flat `Segmented` row to a grouped picker touches `GraphControls.tsx` and its render test. The hand-rolled Sugiyama and Louvain forgo the convenience of mature libraries in exchange for dependency purity — they must be carefully bounded (D6) and seeded (D9) to stay deterministic and within budget. The hierarchical/lineage distinction (D3) is a UX subtlety that must be communicated in the chrome labels and hints.

**Risks.** A future agent could be tempted to add the d3-dag optimal strategies "for prettier crossings" — D6 forbids this and the plan must encode it as a guard, not a convention. A future community-mode tweak could reach for the grouped-force idiom for "organic" feel and silently break determinism — D9 fences this. Re-adopting `graphology` for Leiden later would re-open the retired-dependency wound; if Leiden is ever wanted it must clear a fresh dependency decision, not ride in on Louvain.

**Pathways opened.** A six-mode deterministic catalog over one model gives the product a full data-science layout toolkit ("Obsidian-plus") with a uniform seam, so a seventh mode (e.g. a matrix or arc-diagram projection) is again one union member, one case, one module, one picker entry. The grouped picker scales to the deferred `semantic` mode and any future gated entry via the existing `available` flag.

## Alternatives considered

- **Adopt d3-dag for hierarchical (heuristic strategies, tree-shaken).** Rejected for v1 (D2): even heuristic strategies bundle `javascript-lp-solver`/`quadprog` unless tree-shaking is reliably verified, and the hand-roll generalizes code already owned. Kept as the documented fallback if optimal crossing minimization is later justified.
- **Hierarchical replaces or subsumes lineage ("richer lineage").** Rejected (D3): erases the lineage mode's PROV-convention, holding-lane, and dangling-stub honest-degradation semantics that the representation ADR committed to for the review path-following task.
- **Re-adopt `graphology` + `graphology-communities-louvain` for community detection.** Rejected (D8): re-introduces the dependency the stability ADR deliberately retired, for a single algorithm a ~150-line hand-roll covers at the bounded scale.
- **Grouped-force community placement (membership as a solver attraction force).** Rejected (D9): re-introduces the live solver and breaks the deterministic-seed contract, sacrificing mode-switch object constancy and mental-map stability.
- **Hand-roll radial tidy-tree x-assignment instead of d3-hierarchy.** Rejected (D4): the one place a battle-tested zero-dep ISC library clearly beats a hand-roll, with no purity cost.
- **Gate the new modes behind measurable triggers like `semantic`.** Rejected (D10): there is no unmet measurable condition or wire amendment to gate on; the gate is for genuinely-blocked modes only.
- **Keep the flat `Segmented` row for all modes.** Rejected (D11): a seven-wide flat row is illegible and loses the spatial-vs-temporal distinction the current code already flags.

## Constraints & rule compliance

- **`graph-compute-is-cpu-gpu-is-render-and-search`** — every layout is CPU compute in a framework-free scene module producing world coordinates the GPU only draws; no layout runs on the GPU, none enters the engine. Mirrors `lineageLayout`/`circularLayout`. Compliant.
- **`graph-queries-are-bounded-by-default`** — layouts run only over the LOD-bounded served slice (hundreds of nodes); D6 forbids the super-linear strategies so the layout-time budget is bounded the same way the wire is. Compliant.
- **`published-wheel-purity`** — frontend npm only; no Python wheel/torch/rag concern. The analog discipline (bundle weight + no re-adopted retired lib) is honored: one ISC zero-dep add (D4), two hand-rolls (D2, D8), `graphology` not re-adopted (D8), dead `sigma` removed (D12).
- **`dashboard-layer-ownership`** — all compute in `frontend/src/scene/field/`; chrome only writes the active mode to the view store (`setRepresentationMode`); nothing fetches the engine or reads `tiers`. The new modes ride the existing `set-representation-mode` command (no new command). Compliant.

## Open questions deferred to the plan

- Concrete crossing-reduction iteration count and convergence cutoff for the hand-rolled Sugiyama barycenter sweep (bounded, but the exact cap is a tuning choice for the plan).
- Cycle-removal rendering: do reversed back-edges (D7) render with a distinct marker, or silently? The plan decides whether reversed edges get a visual treatment.
- Community count sanity: whether to cap or merge very small communities for placement legibility, and the exact two-level placement geometry (outer circle radius scaling, inner pack spacing).
- Radial per-component sector allocation: equal sectors vs salience-weighted sector arcs; ring radius scaling per component size.
- Exact grouped-picker component (D11): reuse a kit primitive vs a new grouped variant of `Segmented`, and the precise label/hint copy per mode. NOTE: the live chrome is `frontend/src/app/stage/GraphControls.tsx` (`LayoutGroup`/`Segmented`), not `RepresentationModePanel.tsx` as the research cited — the plan must target `GraphControls.tsx` and reconcile/retire any unused `RepresentationModePanel`.
- Whether the hand-rolled Sugiyama's longest-path code is *extracted* from `lineageLayout.ts` into a shared helper or *duplicated* (the lineage version carries spine/dangling semantics the hierarchical version does not need).
- Test strategy for determinism: golden-position assertions per layout over a fixed bounded fixture, mirroring the existing layout tests.
