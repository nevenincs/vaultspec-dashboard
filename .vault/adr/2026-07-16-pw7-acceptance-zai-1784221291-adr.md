# `pw7-acceptance-zai-1784221291` adr: `SSE reconnection and cursor-persistence strategy for long-lived dashboard event streams` | (**status:** `accepted`)

## Problem Statement

Dashboard event streams power live state synchronization across multiple planes: graph deltas, backend signals, git status, and authoring lifecycle events. These streams must survive transient network blips, browser tab lifecycle changes, and long-running sessions without silent data loss or unbounded memory growth. As documented in `2026-07-16-pw7-acceptance-zai-1784221291-research`, the current implementation has solid foundations—per-scope monotonic sequence numbers, ring-buffer replay with explicit gap events, and exponential-backoff retry—but lacks cross-refresh cursor persistence and unified recovery coordination. Users experience full re-keyframe on browser refresh, and the three independent stream families (graph, backend signals, authoring) retry without coordination.

## Considerations

Key factors from the research anchor (`2026-07-16-pw7-acceptance-zai-1784221291-research`):

- **Three independent stream families** exist with separate retry logic, cursors, and gap handling: graph delta stream (per-scope, `since=keyframeSeq`), backend signals stream (always-on, hidden-tab pause per D4 of `2026-07-11-universal-data-loading-adr`), and authoring lifecycle stream (`last_seq` cursor with recovery snapshot per `2026-06-29-agentic-streaming-events-outbox-adr`).

- **Cursor loss occurs** on browser refresh, TanStack cache eviction, navigation away/back, and server-side ring buffer eviction. All cursors are runtime-only; authoring `authoringStreamCursor` is module-scoped state.

- **Memory bounds are enforced**: `STREAM_RETENTION = 256` chunks, `LIFECYCLE_REPLAY_PAGE_CAP = 128`, `GENERATION_CHANNEL_FRAME_CAP = 256`. Unbounded retention is not viable.

- **Exponential backoff exists** (250ms → 30s capped) with no maximum attempt count. Independent retries across streams could cause thundering-herd behavior on reconnection.

- **EventSource limitations**: browser native reconnection cannot resume with custom `since=` parameter; explicit close/reopen required.

## Constraints

- **TanStack Query cache** is the sole runtime state mechanism; cache eviction (`gcTime: 30_000` on streams) can discard cursors if unobserved.

- **Ring buffer size on engine** is bounded by design; replay window is finite. Server-side gap events (`oldest_buffered` exceeding `since`) force re-keyframe.

- **Authoring generation channels** are non-authoritative and bounded per `2026-06-29-agentic-streaming-events-outbox-adr`; cursor continuity matters only for lifecycle events.

- **Backend signals stream** has no cursor semantics (current-state only per D4 of `2026-07-11-universal-data-loading-adr`); persistence applies only to graph and authoring streams.

- **LocalStorage capacity** is ~5-10MB per origin; cursor persistence must be minimal and scoped to avoid quota exhaustion.

## Considered Options

### Option A: Do Nothing (Status Quo)

Accept full re-keyframe on browser refresh and independent stream retries. Rationale for rejection: users lose scroll position, view state, and recent activity on refresh; thundering-herd retries (all three streams reconnecting simultaneously) could overwhelm the engine; silent re-keyframe masks data loss.

### Option B: LocalStorage Cursor Persistence with Unified Reconnection Coordinator

Persist `lastSeq`/`since` cursors to localStorage for graph and authoring streams; introduce a `StreamCoordinator` that serializes reconnection attempts with jittered backoff; standardize gap recovery UX across all stream types. Rationale for acceptance: balances continuity gains with implementation complexity; localStorage is sufficient for minimal cursor data; coordinator prevents thundering-herd retries; unified gap handling provides consistent user experience.

### Option C: WebSocket Migration with Protocol-Level Cursor Continuation

Replace SSE with WebSocket to enable protocol-level cursor continuation and bidirectional recovery signaling. Rationale for rejection: SSE is entrenched across engine and frontend; WebSocket introduces significantly higher complexity (connection lifecycle, frame parsing, bidirectional protocol); EventSource limitations are workable via explicit close/reopen; no evidence in codebase that WebSocket was evaluated or preferred.

### Option D: Service Worker Background Sync

Implement a service worker to persist cursors and handle background reconnection independent of tab lifecycle. Rationale for rejection: adds substantial complexity (registration, message passing, storage synchronization); dashboard runs in standard SPA context; service worker lifecycle adds debugging surface; localStorage persistence per Option B achieves the primary continuity goal.

## Implementation

### Phase 1: LocalStorage Cursor Persistence

**Persist graph delta cursor** (`frontend/src/stores/server/queries/streams.ts`):
- On every stream event, write `scope:lastSeq` to `localStorage` key `vaultspec:sse:graph:{scope}`
- On stream initialization, read from localStorage; if present, resume with `since={lastSeq}`
- Handle localStorage unavailability (private browsing, quota exceeded) by falling back to zero cursor

**Persist authoring lifecycle cursor** (`frontend/src/stores/server/authoring/index.ts`):
- On `lastSeq` update, write to `localStorage` key `vaultspec:sse:authoring`
- On stream initialization, read from localStorage; if present, resume with `last_seq={cursor}`
- Coordinate with existing `authoringStreamCursor` module state; localStorage is source of truth on mount

**Scope and cleanup**:
- Prefix all keys with `vaultspec:sse:` to avoid collision
- Implement cleanup on workspace switch or explicit clear command
- Validate cursor range on read (normalize to safe integer per existing `normalizeAuthoringStreamSeq`)

### Phase 2: Stream Reconnection Coordinator

**Introduce `StreamCoordinator`** (`frontend/src/stores/server/queries/streamCoordinator.ts`):
- Singleton orchestrator that serializes reconnection attempts across all three stream families
- Implement jittered exponential backoff: `base_delay * 2^attempt + random(0, base_delay/2)`
- Enforce maximum concurrent reconnections (cap at 1 to prevent thundering herd)
- Emit coordinated state events (`reconnecting: {stream, delay}`, `recovered: {stream}`, `failed: {stream, error}`)

**Integrate with existing streams**:
- Graph stream: register with coordinator on mount, honor `pause()`/`resume()` commands
- Backend signals stream: register with coordinator, respect D4 hidden-tab pause grace period
- Authoring stream: register with coordinator, coordinate its existing retry logic

**Backpressure signaling**:
- If engine returns `gap` event with `lagged` count exceeding threshold, coordinator applies exponential backoff before retry
- If multiple streams fail simultaneously, coordinator serializes retries with staggered delays

### Phase 3: Unified Gap Recovery UX

**Standardize gap handling** across all stream types:
- Define `GapRecoveryMode`: `auto` (attempt replay with `since=oldest_buffered`), `manual` (show user notification with "Reload" action), `accept` (log and continue, for non-critical streams)
- Graph stream: `auto` mode with user-facing toast on large gaps (>1000 sequences missed)
- Backend signals stream: `accept` mode (current-state stream, gaps are acceptable per D4)
- Authoring stream: `manual` mode with recovery flow already implemented

**Gap notification component** (`frontend/src/app/chrome/GapNotification.tsx`):
- Display non-intrusive banner when gap detected
- Show "Resume from {oldest_buffered}" or "Reload from beginning" actions
- Dismiss on user action or successful recovery

## Rationale

Option B is chosen because it addresses the core gaps identified in `2026-07-16-pw7-acceptance-zai-1784221291-research` with proportionate complexity:

- **LocalStorage persistence** provides cross-refresh continuity for cursors without server-side changes. Minimal storage footprint (two integers per stream) avoids quota risk. Fallback to zero cursor on unavailability preserves current behavior.

- **Unified reconnection coordinator** prevents thundering-herd retries (all three streams reconnecting simultaneously after network restore) without disrupting existing retry logic. Jittered backoff adds randomness to avoid synchronization. Maximum concurrent reconnection cap of 1 is conservative; can be relaxed if measurement shows headroom.

- **Standardized gap recovery UX** provides consistent user experience across stream types. Per-stream modes respect existing semantics: backend signals gaps are acceptable (D4), authoring gaps require manual recovery (existing flow), graph gaps auto-recover with notification for transparency.

Option A (do nothing) is unacceptable because silent re-keyframe masks data loss and undermines user trust. Option C (WebSocket) is disproportionate migration cost for incremental continuity gains. Option D (Service Worker) adds complexity beyond the core requirement.

## Consequences

### Gains

- **Cross-refresh continuity**: users retain scroll position, view state, and recent activity on browser refresh
- **Reduced server load**: re-keyframe from zero is avoided for short refresh intervals; ring buffer replay serves most cases
- **Consistent UX**: unified gap handling provides predictable recovery behavior across all streams
- **Thundering-herd prevention**: coordinator serializes reconnection attempts, reducing engine spike risk on network restore

### Difficulties

- **LocalStorage edge cases**: private browsing mode rejects writes; quota exhaustion (unlikely with minimal data) requires fallback path
- **Cursor staleness**: persisted cursor may reference sequence outside server replay window; gap detection already handles this via explicit `gap` events
- **Coordinator complexity**: singleton state machine adds debugging surface; must be thoroughly tested for race conditions (e.g., rapid tab hide/show cycles)
- **Multi-tab coordination**: each tab maintains independent localStorage entries; no cross-tab signaling (addressed in uninvestigated areas of research)

### Opens

- **Multi-tab coordination**: if user opens dashboard in multiple tabs, each tab independently persists cursors and reconnects; future work could add BroadcastChannel API for tab activation signaling
- **Cursor versioning**: if engine sequence generation scheme changes, migration path needed for persisted cursor format
- **Long-running session limits**: no maximum session duration; future work could add forced reconnection interval to bound memory growth

### Supersession

This ADR supersedes no existing decisions. It extends the hidden-tab pause decision (D4 of `2026-07-11-universal-data-loading-adr`) by adding persistence and coordinator for always-on streams. It complements the durable lifecycle events architecture (`2026-06-29-agentic-streaming-events-outbox-adr`) by providing client-side cursor continuity for the same recovery semantics.
