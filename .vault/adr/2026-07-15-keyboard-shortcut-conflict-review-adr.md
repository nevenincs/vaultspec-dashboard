# `keyboard-shortcut-conflict-review` adr: `one scope-aware conflict definition, a platform-reserved denylist, and three cross-platform chord fixes` | (**status:** `accepted`)

## Problem Statement

The Settings keyboard-shortcuts recorder (`app/settings/controls/KeybindingControl.tsx` via `keybindingConflictPresentations` in `stores/view/settingsControls.ts`) flags ten rows across five deliberate global-vs-canvas shadow pairs (`ArrowLeft/Right/Up/Down`, bare `E`) as conflicting on a stock install with zero user overrides ŌĆö violating the user's law that DEFAULT shortcuts must never conflict. The root cause is that the recorder's conflict predicate, `conflictsForCandidate` (`platform/keymap/registry.ts`), flags any chord collision whose contexts merely overlap (`contextsOverlap`, where global overlaps everything), with no awareness of the dispatcher's own most-specific-context-wins resolution (`resolveKeybinding`). The static default-set guard (`stores/view/defaultKeybindingConflicts.guard.test.ts`, built to close KAR-008 of the keyboard-action correctness audit) already knows the correct, narrower definition ŌĆö it locally re-implements a specificity filter over `findConflicts` ŌĆö but that correct definition lives only in the test file and is never consumed by the recorder the user actually sees. Runtime resolution is correct; two of the three surfaces that describe it to a human are wrong or silent.

Independently, this review's cross-platform sweep found three live defects unrelated to the conflict-definition bug: the right-rail tab chords `Mod+1`/`Mod+2` (`stores/view/rightRailKeybindings.ts`) are dead on arrival ŌĆö every major browser hard-reserves `Ctrl`/`Cmd+1..9` for browser-tab switching and intercepts the keystroke before it reaches the page; the search-palette chord `Mod+P` (`stores/view/commandPalette.ts`) sits on browser Print with inconsistent preventability across Firefox/Safari, unlike this codebase's own documented practice of avoiding such chords (`Mod+R`ŌåÆ`Mod+Shift+R` in `reloadKeybindings.ts`, `Mod+W`ŌåÆ`Mod+Alt+W` in `editorKeybindings.ts`); and the chord model (`platform/keymap/chord.ts`) has no compensation for AltGr, which on Windows/EU keyboard layouts emits `ctrlKey+altKey` ŌĆö indistinguishable from `Mod+Alt+ŌĆ”` on non-Mac ŌĆö colliding with the live `Mod+Alt+[`/`Mod+Alt+]` defaults (`leftRailKeybindings.ts`) on exactly the layouts where those bracket characters themselves require AltGr to type.

This ADR fixes the architecture of "what counts as a conflict" and closes the concrete cross-platform defects the review found; it does not replace or re-architect the keymap registry the keyboard-action-system ADR built.

## Considerations

- **Two definitions of conflict must become one (research finding).** The dispatcher's runtime truth (`resolveKeybinding`, most-specific-context wins) and the recorder's displayed truth must agree, or the UI keeps lying about behavior that is actually correct. The definition must also keep flagging genuinely bad user overrides ŌĆö two global bindings, or two bindings inside the same surface context, sharing a chord ŌĆö where the dispatcher's id-order tie-break really does silently shadow one action (KAR-008's original finding, still real).
- **The guard's specificity filter is currently local, not shared.** `defaultKeybindingConflicts.guard.test.ts` re-implements `specificityRank` itself rather than calling a predicate the production recorder also calls. A fix that only patches the recorder (or only the guard) re-creates the same two-definitions problem in a different shape.
- **A re-chord is not "any free chord."** The codebase has a visible chord-family convention: `Mod+Alt+*` for structural/window actions, `Mod+Shift+*` for the "safe variant of a browser-reserved chord" family (`Mod+Shift+R`, `Mod+Shift+O`, `Mod+Shift+G`, `Mod+Shift+X`, `Mod+Shift+F`, `Mod+Shift+L`), with inline comments citing the specific browser reservation each avoidance addresses. A replacement chord must land inside one of these families and carry the same style of reservation comment.
- **AltGr is a real, live defect, not a hypothetical.** `Mod+Alt+[`/`Mod+Alt+]` are shipped defaults; on a German (or most EU) Windows layout, AltGr combinations producing `[`/`]` emit `ctrlKey=true, altKey=true` ŌĆö the exact bit pattern `matchesChord` treats as `Mod+Alt` on non-Mac. A full fix (per-layout detection or a physical-code matcher) is a chord-model redesign outside a conflict-cleanup ADR's scope; a guardrail plus a return trigger is scoped instead.
- **Bare-key global bindings (`E`, `Backspace`, `?`) already have a text-entry gate** (`handleKeymapEvent` in `stores/view/keymapDispatcher.ts`). The one confirmed gap is IME composition: `event.isComposing` is never checked, so a composition-confirming keystroke inside an active IME candidate window can still resolve to a global binding.
- **Relationship to priors.** This ADR narrows and amends keyboard-action-system decision 7 (the shared conflict-check becomes scope-aware) and extends the correctness audit's KAR-008 disposition to also cover the recorder surface and a new platform-reserved denylist. It does not touch the registry's one-chord-per-action model, the ActionDescriptor/dispatcher plane, or the FocusZone/region-cycling work.

## Considered options

- **O1 Re-chord broadly to eliminate every shared chord** ŌĆö rejected as a general strategy: it would re-chord the deliberate, resolvable global-vs-canvas arrow/`E` shadows, a UX regression (arrows-walk-the-graph-on-canvas is desired behavior), not a fix. Applied narrowly (D4/D5) only where a chord is dead or unreliable regardless of context.
- **O2 One formal, scope-aware conflict definition, shared by recorder and guard ŌĆö CHOSEN.** Fixes the reported bug at its root with zero UX cost and keeps genuine-collision detection.
- **O3 Platform-reserved denylist + guard ŌĆö CHOSEN, additive to O2.** O2 only tells you two bindings collide with each other; it says nothing about a single binding colliding with the browser/OS. A `Mod+1` binding has no in-app conflict yet is completely dead. A denylist is the only mechanism that catches this class, in CI, before a future default ships broken.
- **O4 Per-platform default chord maps (VS Code-style)** ŌĆö deferred. The three concrete defects are each solvable with a single stable re-chord or a documented scope-out; none requires per-platform branching. Revisit only if a chord proves genuinely unsatisfiable on every platform simultaneously (return trigger in Non-goals).

## Constraints

- **Frontend-only.** The engine-owned `keybindings` setting (sparse override map, validated chords, size cap) is untouched ŌĆö only which chords are DEFAULTS and how the frontend classifies a collision change. No wire-shape change, no storage migration.
- **No new runtime dependency.** The scope-aware predicate, the denylist, and the composition check are pure additions to existing small TypeScript modules ŌĆö consistent with the parent ADR's no-keybinding-library precedent.
- **Bounded by default.** The denylist is a fixed, small, hand-maintained array (no accumulator, no runtime growth); the existing override bounds are untouched.
- **No deprecation bridges.** The recorder's old overlap-only semantics and the guard's local `specificityRank` duplicate are deleted outright once both consume the shared predicate.

## Implementation

**D1 ŌĆö One formal, scope-aware conflict definition, exported from the registry as the single source of truth.** `platform/keymap/registry.ts` gains the specificity-equality filter that today lives only inside the guard test: a `specificity(context)` helper (mirroring the private one in `resolveKeybinding`) is exported, and `findConflicts`/`conflictsForCandidate` exclude a pair unless both contexts sit at EQUAL specificity. A **conflict** is formally: two distinct action ids whose effective chords canonicalize identically AND whose binding contexts are at equal specificity (both global, or both the same named surface context). A global-vs-surface pair at differing specificity is by definition NOT a conflict ŌĆö the dispatcher's most-specific-active-context-wins rule resolves it deterministically. This is the one place the rule is stated; every surface consumes it from `registry.ts`, never re-derives it.

**D2 ŌĆö The recorder consumes D1's definition; the guard test consumes it too, deleting its local duplicate.** `stores/view/settingsControls.ts` needs no call-site change ŌĆö it already calls `conflictsForCandidate`, so the fix propagates once the registry predicate narrows. The guard test deletes its local `specificityRank`/manual filter and asserts directly against the now-scope-aware `findConflicts`. Verified outcome: on a stock install, `keybindingConflictPresentations` returns empty for all ten previously-flagged rows; a genuinely bad user override (two globals, or two same-surface bindings, on one chord) still surfaces the inline warning.

**D3 ŌĆö A platform/browser-reserved denylist, enforced by a new guard.** New module `platform/keymap/reservedChords.ts`: a small, flat, hand-maintained array of canonicalized chords hard-reserved by a major browser or OS regardless of preventDefault ŌĆö `Mod+1`ŌĆ”`Mod+9` (browser-tab switching), `Mod+W`/`Mod+T`/`Mod+N`/`Mod+Q` (codifying the existing avoidance practice), macOS `Cmd+H`/`Cmd+M`/`Cmd+Q`, and (per D5) `Mod+P`. Each entry carries a comment naming the reservation and browsers/OS, matching the codebase's existing convention. New `reservedKeybindingDenylist.guard.test.ts` asserts no canonicalized default chord in the assembled default set appears in the denylist. Maintenance: append-only, an entry added only on an independently confirmed reservation.

> **Amendment (2026-07-15, review-caught): D3's denylist gains the chrome-level `Mod+Shift+*` class.** The review that re-chorded D5 surfaced that several `Mod+Shift+*` chords are also browser-reserved regardless of `preventDefault`, so the denylist adds `Mod+Shift+P` (Firefox New Private Window), `Mod+Shift+N` (incognito/InPrivate), `Mod+Shift+T` (reopen tab), `Mod+Shift+W` (close window), and the devtools openers `Mod+Shift+I`/`Mod+Shift+J`/`Mod+Shift+C`/`Mod+Shift+K`, each with a per-entry citation. No current default collides with any addition (the denylist guard confirms).

**D4 ŌĆö Re-chord the right-rail tab chords off the dead `Mod+1`/`Mod+2` to `Mod+Alt+1`/`Mod+Alt+2`**, landing in the existing structural `Mod+Alt+*` family (beside the left-rail and doc-tab chords). A reservation problem, not a semantics problem ŌĆö a straight re-chord in the established family, with D3's guard proving the dead chords never reappear.

**D5 ŌĆö Re-chord `Mod+P` off browser Print to `Mod+Shift+P`**, landing in the existing `Mod+Shift+*` safe-variant family. This applies the standard the codebase already applies to `Mod+R`/`Mod+W` ŌĆö "cannot be reliably prevented in every major browser" disqualifies a default ŌĆö consistently, with the same reservation-comment convention. `Mod+P` joins D3's denylist so it cannot be reused as a default.

> **Amendment (2026-07-15, review-caught): D5's chosen chord is `Mod+Alt+S`, not `Mod+Shift+P`.** Code review found `Mod+Shift+P` is Firefox's hard-reserved New Private Window (`Ctrl/Cmd+Shift+P`), dead on arrival there ŌĆö the same defect class as the original `Mod+P`. The `Mod+Shift+*` family is not the "safe variant" family it appeared to be: several of its members are chrome-level (`Mod+Shift+K` Firefox Web Console, `Mod+Shift+N` incognito, `Mod+Shift+T` reopen-tab, `Mod+Shift+W` close-window, plus the devtools openers). The P mnemonic could not be kept in the structural `Mod+Alt+*` family either: `Mod+Alt+P` is already the `project:browse` default. Search therefore binds `Mod+Alt+S` ŌĆö a free letter chord in the codified `Mod+Alt+*` structural family (no AltGr hazard, no in-app collision), verified by the conflict and denylist guards. Both `Mod+P` and `Mod+Shift+P` remain on the D3 denylist so neither returns as a default.

> **Amendment (2026-07-15, review-round): `app:document-search` re-chorded off `Mod+Shift+O` onto `Mod+Alt+F`.** The same review round found `Ctrl/Cmd+Shift+O` is Chrome's Bookmark Manager and Firefox's Library ŌĆö chrome-level and page-uninterceptable, the same dead-on-arrival class D5 addresses. Document-find now binds `Mod+Alt+F` ("find in documents"), a free letter chord in the structural `Mod+Alt+*` family; `Mod+Shift+O` joins the D3 denylist. Sweep of the remaining `Mod+Shift+*` defaults against the chrome-level reservation class: `Mod+Shift+R` (hard reload, page-preventable ŌĆö deliberate, documented), `Mod+Shift+X`, `Mod+Shift+F`, `Mod+Shift+L` verified clean (no chrome-level reservation); `Mod+Shift+G` overlaps Firefox find-previous (page-level, preventable ŌĆö clean). `Mod+Shift+D` (editor draft-diff toggle, `editor:toggle-diff`) was FOUND to be the same-class defect ŌĆö Chrome/Edge `Ctrl+Shift+D` bookmarks all open tabs, chrome-level and page-uninterceptable ŌĆö and re-chorded onto `Mod+Alt+D`; `Mod+Shift+D` joins the D3 denylist.

> **Amendment (2026-07-15, re-review): editor draft-diff `Mod+Alt+D` -> `Mod+Alt+G`.** The reviewer re-check found `Mod+Alt+D` is macOS `Cmd+Opt+D`, Apple's documented system-wide Show/Hide Dock shortcut ŌĆö the same OS-reservation class as the `Mod+H`/`Mod+M` denylist entries, and not interceptability-verifiable on real Mac hardware. By the campaign's disqualify-on-unreliable-preventability standard it is re-chorded defensively to `Mod+Alt+G`, clean across Chrome/Firefox/Safari/Edge chrome-level and macOS system shortcuts (distinct from the `Mod+Shift+G` graph toggle). Candidates vetted OUT of the `Mod+Alt+*` family in re-review, for durability: `Mod+Alt+B` (Safari `Cmd+Opt+B` bookmarks editor), `Mod+Alt+K` (Firefox mac `Cmd+Opt+K` Web Console), `Mod+Alt+E` (Safari `Cmd+Opt+E` Empty Caches), `Mod+Alt+C` (Chrome mac `Cmd+Opt+C` inspect element), and `Mod+Alt+M`/`Mod+Alt+H` (same Dock/window OS class as D). `Mod+Alt+D` joins the D3 denylist beside `Mod+H`/`Mod+M` with the Apple Show/Hide Dock citation.

**D6 ŌĆö Close the IME composition gap on bare-key global bindings.** `handleKeymapEvent` gains an unconditional early-out ŌĆö before context resolution, independent of the text-entry gate ŌĆö when `event.isComposing === true` (with the legacy `keyCode === 229` fallback). Composition can occur inside contentEditable regions the text-entry target check does not recognize, so gating on `isComposing` directly covers `E`/`Backspace`/`?` and every other binding uniformly.

**D7 ŌĆö AltGr: explicitly out of scope, with a guardrail and a return trigger.** No per-layout or physical-keycode disambiguation lands now (a chord-model redesign, closer to O4). Instead: (1) a documented convention in `platform/keymap/chord.ts`'s module comment ŌĆö a new DEFAULT chord must never be `Mod+Alt+<symbol>` where the symbol requires AltGr on a common EU layout (bracket/brace/at/euro-class) ŌĆö reviewer-enforced, since "requires AltGr" is layout-dependent and not derivable from the DOM key value; (2) the two live instances (`Mod+Alt+[`/`Mod+Alt+]`, left-rail tree expand/collapse) are flagged in `leftRailKeybindings.ts` as a known accepted risk pointing at this decision, with a return trigger: on a real affected-layout report, re-chord those two specifically (e.g. toward `Mod+Shift+[`/`]`) rather than attempting a general AltGr fix.

**D8 ŌĆö Amends keyboard-action-system D7 and KAR-008.** The parent decision's "one pure conflict-check both surfaces use" becomes scope-aware (what it always intended but did not specify); KAR-008's guard is extended in place ŌĆö its filter relocates into production code, the recorder consumes the same predicate, and the sibling denylist guard covers the collide-with-the-browser class KAR-008 never asked about.

## Non-goals

- **O4 (per-platform default chord maps) stays deferred.** No chord in this review requires per-platform branching. Return trigger: a future chord proven unsatisfiable simultaneously on every target platform.
- **The `?` legend is unchanged.** It derives correctly from the registry and overrides, was never the buggy surface, and picks up the re-chords transparently.
- **No engine change.** The `keybindings` setting's shape, validation, and cap are untouched.
- **No general AltGr/physical-keycode remediation** (D7) and no full `Mod+Alt+*` family re-chord ŌĆö only the two symbol-key instances are flagged.

## Rationale

The research is decisive that this is two independent, narrow problems wearing one report: a definitional bug (the recorder disagrees with the dispatcher about what "conflict" means) and a handful of concrete cross-platform chord defects. O2+O3 fix the definitional bug at its root ŌĆö one shared predicate ŌĆö rather than patching the recorder's symptom, which would let the guard and recorder drift apart again; this is the same one-source-of-truth pattern the parent ADR established for the registry itself. The denylist is the only mechanism that catches a `Mod+1`-class defect, because the browser is the collision ŌĆö there is nothing in-app to detect. The re-chords land inside the codebase's own chord-family conventions so the legend and recorder show a coherent scheme, not an ad hoc patch. D6 (IME) and D7 (AltGr) are treated asymmetrically on purpose: D6 is a single well-defined boolean check with no design risk, so it lands now; D7's proper fix needs layout research this ADR has not done, so it is scoped out honestly with a guardrail and a trigger rather than shipped as a guess.

## Consequences

- **Gains.** The Settings recorder shows zero false-positive conflicts on a stock install while still catching genuinely bad user overrides; a CI guard proves no default chord is browser/OS-dead; the two DOA right-rail chords and the unreliable print-adjacent search chord become reliable across Chrome/Firefox/Safari/Edge; the IME composition gap closes for every binding at one gate; the conflict definition and the denylist are each authored once and consumed everywhere, closing the drift class KAR-008 first named.
- **Costs.** Two default-chord changes are user-visible (`Mod+1`/`Mod+2` never worked, so no muscle memory breaks; `Mod+P` worked in Chrome/Edge ŌĆö the one real behavior change, worth a release-note line). The denylist is hand-maintained with no machine source of truth for "really reserved" ŌĆö mitigated by requiring an independently confirmed citation per entry.
- **Pitfalls.** D7's guardrail is reviewer-enforced, not test-enforced ŌĆö a future `Mod+Alt+<symbol>` default has no mechanical catch; flagged plainly so it is not mistaken for guarded. A future contributor extending `findConflicts` without understanding D1's specificity filter would change the predicate for every consumer at once ŌĆö the module comment states this.
- **Pathways.** The denylist module is a natural home for future platform reservations; the scope-aware `findConflicts` is reusable by any future surface reasoning about bindings (e.g. warning on a risky custom binding before persisting).

## Verification strategy

- **D1/D2** ŌĆö the conflicts guard test (rewritten to the shared predicate, local duplicate deleted) proves the default set collision-free; settings-control tests assert `keybindingConflictPresentations` returns empty for each of the ten previously-flagged rows on a stock override map, and still fires for a synthetic same-specificity override collision.
- **D3** ŌĆö new denylist guard asserts the assembled default set contains no denylisted chord, plus a synthetic inject-`Mod+1` case proving the guard has teeth.
- **D4/D5** ŌĆö keybinding and palette test expectations update to `Mod+Alt+1`/`Mod+Alt+2` and `Mod+Shift+P`; the denylist guard proves the old chords are gone from defaults.
- **D6** ŌĆö a dispatcher test drives `handleKeymapEvent` with `isComposing: true` on a bound bare-key chord and asserts no action fires.
- **D7** ŌĆö no automated test (documented, reviewer-enforced); the `chord.ts` module-comment guardrail and the `leftRailKeybindings.ts` inline note are the durable artifacts.
- **Full gate** ŌĆö `just dev lint frontend` green; the existing keymap/navigation suites re-run unchanged to confirm the Class A/B split and canvas-shadow behavior are preserved.
