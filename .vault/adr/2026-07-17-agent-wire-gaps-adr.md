# `agent-wire-gaps` adr: `engine wire-gap closure for the authoring agent plane` | (**status:** `accepted`)

## Problem Statement

The agentic-authoring-ux epic (W01–W04, shipped) built the docked Agent panel, composer, live transcript, comment→agent bridge, and autonomy control against the authoring backend as it exists. During that build, six served-shape gaps were test-proven and recorded as dated amendments in the agentic-authoring-ux ADR, each closed frontend-side with an honest interim (client-staged records, "session-actor-latest" correlation, disabled-with-reason pills) and a filed cross-team ask. Those asks are now OURS to close engine-side. A seventh gap rides along: the agentic-feedback-loop ADR's D4 structured-feedback continuation, whose engine half was never built (comments ride in the prompt as prose today).

The gaps, in the amendments' priority order: (1) no run-settle/completion lifecycle event — a run stays `active` until cancelled, so "Done" can never render; (2) run cancel destroys the whole session, and there is no real queued-turn primitive (a mid-run turn silently JOINS the active run); (3) no served pending-interrupt state and no typed decision schema (the only surface is the tool-execute `awaiting_permission` arm's `interrupt_id`); (4) `session_id` is stored on the changeset but omitted from `ProposalProjection`, and run/turn provenance does not exist on a changeset at all; (5) operation mode is writable via `POST /v1/mode` but has no read route; (6) no served model options and no model parameter anywhere in the session contract.

This ADR decides the wire shape of each closure — route, event kind, field, bounded vocabulary, resource bound, and compat posture — and names what each unlocks in the shipped frontend. The a2a relay channel (the separate `/ops/a2a/*` edge, built externally) is explicitly out of scope; where a gap was provisionally "gated on a2a," this ADR decides whether an engine-native closure is right anyway.

## Considerations

- **The engine already owns most of the vocabulary.** `RunStatus` carries `Active | CancelRequested | Cancelled | Completed | Failed` and `RunRecord.completed_at_ms` exists — but no code path transitions a run to `Completed`/`Failed` and no lifecycle event announces it. The mode store already answers `current_record(scope_id)`; only the HTTP read is missing. The changeset revision already stores `session_id`. Most closures are exposure, not invention.
- **The agent runtime is external to the engine.** LangGraph execution lives behind an adapter that only maps thread/run/checkpoint references onto authoring records; the a2a sibling drives real runs from outside. The engine therefore cannot OBSERVE a run finishing — it can only be TOLD. Any completion design must be a reported command from the party driving the run, not an engine-side watcher.
- **The tool executor already knows the run.** `ExecuteToolCallRequest` carries `run_id` at dispatch time, so stamping run/turn provenance onto a proposal created through the `propose_changeset` tool is plumbing an in-hand value, not new correlation machinery.
- **House rules bind every closure:** each response rides the shared tiers envelope (wire-contract rule); every list is capped and every queue bounded at creation (resource-bounds rule); the authoring module owns durable workflow state in its store but never hand-writes `.vault/` documents (architecture-boundaries rule); interim frontend affordances are deleted at cutover, never bridged (no-deprecation-bridges doctrine).
- **Stable-key discipline:** `ProposalProjection` is not an edge/node stable-key surface and `changeset_id` is client-supplied, so adding served provenance fields to the projection and record changes no identity composition. Provenance fields naming the producing fact are exactly the class the wire-contract rule admits.
- **Frontend interims name their own cutovers.** The one-slot client queue, the client-staged interrupt annex, the "session-actor-latest" correlation mark, and the proposal-gated autonomy control are all built to be deleted the moment the served shape lands — the epic ADR commits to that disposal.
- **The authoring store is versioned SQLite** with migration discipline (new tables/columns land as a new schema version; a changed migration at the same version requires a clean rebuild of dev stores).

## Considered options

- **O1 — Defer everything to the a2a relay.** Rejected: the relay carries transient token/tool-call frames (a2a D3); run settlement, queued turns, interrupt state, provenance, and mode are DURABLE product truths that belong in the authoring store and its lifecycle feed regardless of transport. Deferring them would make the relay the source of durable truth, which the a2a ADR itself forbids.
- **O2 — Engine-side run watcher that infers completion (e.g. from runtime polling or checkpoint activity).** Rejected: the runtime is behind an adapter that maps references only; inferring settlement would guess state the engine cannot observe and violate the read-and-infer posture with a liveness heuristic.
- **O3 — Runtime-reported settlement command + durable lifecycle event, with run-scoped cancel and an engine-side bounded turn queue — CHOSEN.** The party driving the run reports its terminal state through the same authenticated command plane it already uses for turns and tools; every other gap closes as a read route or a served field over state the store already holds.
- **For the queued turn specifically — keep the client one-slot queue.** Rejected: with settlement durable (D1), the engine knows the exact promotion moment inside its own unit of work; a client queue loses the prompt on reload and races the settle. The JOIN arm is also a silent-data-loss trap (a second prompt vanishes into an existing run).
- **For model options — invent a static engine enumeration.** Rejected: the runtime has no model-selection surface; a served list nothing honors is a fake enumeration. Deferred with a return trigger (D6).

## Constraints

- Every new route is enveloped through the shared response helpers (success and error both carry `tiers`); no hand-built bodies.
- Mutating commands ride the existing `ResolvedCommand` extractor: middleware-resolved principal (never body-claimed), idempotency key, authorization floor, typed denial kinds. Reads stay principal-permissive like every other authoring read.
- Every list read is capped with an honest `truncated` marker; the turn queue is capped at creation; new lifecycle events flow through the existing deduped outbox (already bounded).
- Store changes land as one new authoring schema version (tables/columns additive; existing records deserialize with absent optionals as `None`).
- The a2a edge contract is frozen by its accepted ADR; nothing here adds a2a verbs or changes the pass-through. The relay remains the future transport for token streams; this ADR touches only durable truths.
- Frontend cutovers (queue chip, staged-interrupt annex, correlation mark, proposal-gated mode control) are in-scope consequences but execute in the frontend lanes, not here.

## Implementation

**D1 — Run settlement: a reported terminal transition + one lifecycle event.** New mutating command route `POST /authoring/v1/runs/{run_id}/settle`, body `{outcome, summary?, failure_reason?}` with `outcome` a bounded two-token enum `completed | failed` (`summary` ≤ the existing turn-summary bound; `failure_reason` validated like the cancel reason, ≤ 500 bytes). Semantics: `Active`/`CancelRequested` → `Completed` or `Failed`, `active=false`, `completed_at_ms` stamped (the fields and enum variants already exist — this is the missing transition); an already-terminal run replays its recorded outcome idempotently, exactly like cancel. Authorization: the settling principal must be the run's `owner` (or its delegator per the existing delegation guards) — the party driving the run reports it; nobody else may forge settlement. The transition emits a new `LifecycleEventKind::RunSettled` (`run.settled`) on the authoring SSE feed carrying `{ "run": RunRecord }` — the terminal status rides inside the bounded `RunStatus` vocabulary, so one event kind covers both outcomes; `cancellation.recorded` remains the cancel path's event, unchanged. Who calls it: the langgraph adapter/a2a runtime as its final act of a run; in today's client-driven serial loop the frontend session principal owns the run and settles it when its dispatch loop finishes — both are the same wire contract. One hazard is named rather than papered over: a runtime that dies without reporting leaves its run `active` forever. No new loop ships for that; abandoned-run reaping (an active run whose `updated_at_ms` is older than a bounded `RUN_STALE_AFTER_MS` → `Failed` with a distinct `abandoned` failure reason, emitting the same `run.settled`) is assigned to the ONE bounded, timed background janitor a prior arch-reviewer advisory already proposes — its duties are the genuinely undriven expiry seams (tool-permission expiry, interrupt reaping, lease expiry, and now this abandoned-run reap). Generation-transcript compaction is NOT a janitor duty: it already runs opportunistically inside every prompt-turn unit of work by a deliberate no-background-loop decision, and it stays there; at most the janitor may run a stated BACKSTOP sweep of the same bounded due-set for a session that never receives another turn, never a second owner. This ADR adds the run-reap duty to that single-janitor posture and explicitly does not create a second timer. Unlocks: "Done"/"Failed" renders from the wire; run-settle is the promotion trigger for D2's queue; the transcript's relay-gap seam gets an honest terminal boundary.

**D2 — Run-scoped cancel, explicit session cancel, and a real queued-turn primitive.** Three coupled changes, one cutover, no bridges:
- `POST /v1/runs/{run_id}/cancel` becomes run-scoped: the run → `Cancelled` exactly as today, but the session-cascade (session.status → cancelled) is DELETED. The session stays `Active`; Stop halts the run and the conversation continues. `cancellation.recorded` is unchanged.
- Session termination becomes its own explicit command: `POST /v1/sessions/{session_id}/cancel`, body `{reason}` (same 500-byte validation), cancelling the session AND its active run if one exists, emitting `cancellation.recorded` for the run (when one was cancelled) and a new `session.cancelled` lifecycle kind for the session aggregate. The `start_prompt_turn` 422-on-non-active-session behavior stands — it is correct once cancel no longer collapses the session.
- The mid-run JOIN arm of `start_prompt_turn` is DELETED and replaced by a real queue: a turn submitted while the session has an active run is persisted as a queued `PromptTurnRecord` (no run yet), bounded by `TURN_QUEUE_CAP = 8` per session (a full queue is a typed 422, `authoring_turn_queue_full`); the command outcome status vocabulary gains `queued` (alongside `started`; `joined` is deleted). Promotion is engine-side and atomic: inside the same unit of work that settles (D1) or cancels a run, the oldest queued turn — FIFO by `turn_index` — is promoted into a fresh run, emitting `run.started` exactly as a direct start does. A new `turn.queued` lifecycle kind announces enqueue so the panel renders served queue state; the session snapshot serves the queued turns within its existing capped window (turns without a run join are self-describing as queued — no schema addition beyond the run's absence, plus an explicit `queued_turn_ids` list on the snapshot so the client never infers). Recovery/resume of an in-flight run stays on the existing `resume_run` — the deleted JOIN was never the recovery seam. Unlocks: the client one-slot queue chip is deleted; Stop stops without destroying the conversation; multi-prompt composition becomes honest served state.

**D3 — Served interrupt state + typed decision schema.** The store already answers this: `interrupts_for_run(run_id, cap)` exists (raise-order, bounded, built in the backend epic's interrupt work) with no route over it, and a prior arch-reviewer advisory already proposed exactly `GET /v1/runs/{run_id}/interrupts` as a recovery listing for a client that loses the `/execute` `awaiting_permission` response. This decision SUBSUMES that advisory: same route, same recovery purpose, plus the typed decision schema the advisory did not cover. New principal-permissive read `GET /v1/runs/{run_id}/interrupts` exposing the existing store query as a bounded page (cap `INTERRUPT_LIST_CAP = 50`, raise-order as the store serves it with pending entries flagged, `truncated` marker) of interrupt projections: `interrupt_id`, `run_id`, `kind` (the existing bounded `InterruptKind` vocabulary — `tool_permission` today), `tool_call_id`, `resume_state` (`pending | resolved`), timestamps, and — replacing the opaque stored decision string on the wire — a TYPED per-kind decision projection: for `tool_permission`, `{ "decision": "approve" | "deny", "comment"?: string }`, mirroring the existing `ToolPermissionDecisionRequest` vocabulary so the write and read schemas are one language. The store gains the bounded list query (it has get-by-id only); the durable record keeps its opaque decision column, with the projection parsing it through the typed schema and serving a `decision_unreadable` marker rather than failing the list if an old record predates the discipline. The interrupt-resume write route is unchanged (resume-by-id stands); `InterruptResumeRequest`'s opaque payload is narrowed to the same typed schema in the same cutover. Unlocks: the W03 client-staged interrupt annex is deleted; a reloaded panel recovers pending permission prompts from the wire; steer renders served pending state instead of a client memory.

**D4 — Proposal↔run correlation served.** Two additive changes. (a) `ProposalProjection` gains `session_id?` — a one-field exposure of what the changeset revision already stores. (b) The changeset revision input gains optional `run_id`/`turn_id` provenance, stamped at the one place the producing fact is in hand: the tool-executor dispatch (its request already carries `run_id`; the turn joins through the run record), flowing through `create_proposal` into the ledger record and out through the projection as `run_id?`/`turn_id?`. Human/direct changesets simply carry `None`. Per the wire-contract rule these are provenance fields naming the producing fact — admissible — and no stable-key composition changes: `changeset_id` remains client-supplied and no edge/node key derives from these fields; this posture is recorded here as the reviewed contract event for the record-shape change. Migration: additive optional fields; existing revisions deserialize as `None`. Unlocks: the inline proposal card binds exactly per-run (`data-correlation="run-id"`), retiring "session-actor-latest" matching without touching the reused card.

**D5 — Scope-level operation-mode read.** New principal-permissive read `GET /v1/mode` serving the active workspace scope's `OperationModeRecord` — `mode` in the bounded existing vocabulary (`manual | assisted | autonomous`), `scope_id`, the setting actor, and `updated_at_ms` — straight off the store's existing `current_record` (the default record when none was ever set, exactly as the policy path already resolves it). The write route is unchanged. No new event kind: mode changes already store a mode event; if the feed lacks a consumable kind for it, that is a frontend-adapter concern over the existing event, not a new contract. Unlocks: the autonomy control renders pre-proposal (an empty review queue no longer hides the control), reading the same record the write round-trips.

This decision is also the first record to touch the mode plane since the authoring-backend epic, so it reconciles that epic's three unscheduled mode-plane advisories rather than leaving them adrift:

- *Session narrowing-override wiring* — DEFERRED with a return trigger. The policy math is fully built (narrowing-only resolution, an honest `session_override_ignored` flag on the decision projection) but the tool-execute route hardcodes `session_override: None` because NO session record field holds an override and no surface writes one: the hardcoded `None` is currently the truth, and wiring a parameter nothing can set would ship a permanently-dead seam. Return trigger: the first product surface offering a per-session autonomy choice (a composer/session-settings control). The wiring is then one shape: an optional narrowing-validated `mode_override` on `CreateSessionRequest`, stored on the session record, threaded into the tool-execute dispatch in place of the `None` — the policy plane needs no change.
- *Mode-set re-narrowing on return* — RECONCILED as already-held behavior, now made observable. Effective mode is resolved per policy decision from the store's current record (nothing caches it engine-side), so a scope-mode downgrade re-narrows every session at its very next gated decision. D5's read serves that same record, and any client caching mode must refetch on the mode-change signal. If a future autonomous multi-step run's window is ever long enough that a mid-run downgrade must interrupt rather than merely gate the next decision, that is the return trigger for an active-run re-check — not built now.
- *Direct-write capability admin seam* — DEFERRED, standalone, explicitly out of this ADR's scope. Today the capability is granted at actor registration for the single implicit local operator, which is sufficient and honest. Return trigger: a second registered human principal or any remote principal, at which point who-grants-direct-write needs its own decided admin surface.

**D6 — Model options: DEFERRED, honestly runtime-gated.** The langgraph adapter maps references only; no runtime surface enumerates or accepts a model choice, and a served enumeration nothing honors would be a fake. No route ships. Return trigger: the moment the agent runtime (langgraph adapter or the a2a edge's preset/agent-card surface) exposes model selection, ship `GET /v1/models` (bounded served enumeration) plus an optional `model` field on `CreateSessionRequest`/`StartPromptTurnRequest` validated against it, and the Model pill's disabled-with-reason state is deleted in the same cutover. Until then the pill's "Default" reason remains the truth.

**D7 — Structured feedback on the turn contract (the feedback-loop D4 engine half).** The immutable feedback batch becomes engine state now; only its cross-repo transport stays on the a2a edge. New mutating command `POST /v1/feedback-batches`: snapshots an ordered set of comment ids (cap `FEEDBACK_BATCH_COMMENT_CAP = 32`, matching the shipped composer batch cap) with bodies, anchors, author identity, source revision, session id, optional general instruction — immutable once created, digest-addressed, stable `feedback_batch_id`, bounded total byte size; later comment edits never mutate it (the accepted feedback-loop D3 verbatim). New principal-permissive read `GET /v1/feedback-batches/{feedback_batch_id}` serves the snapshot. `StartPromptTurnRequest` gains `feedback_batch_id?`; the engine verifies existence, session ownership, and revision fences, and records the reference on the `PromptTurnRecord` — so a revision is auditable to the exact batch it consumed. The composer keeps serializing a human-readable comment block into the prompt for the agent's benefit, but the batch rides as DATA beside it; when the a2a edge later carries the continuation cross-repo it transports this same identifier (feedback-loop D4, unchanged as a contract, now with a real referent). Retention: batches register as protected product state while referenced by a turn, compactable after the turn's transcript window otherwise. Unlocks: comments attach as data, not prose; agent runs become batch-auditable; the a2a ask shrinks from "invent the primitive" to "carry the id".

**Sequencing note (not a plan):** D1+D2 are one coupled store/lifecycle change (settlement is the queue's promotion trigger) and land first; D3, D4, D5 are independent exposures; D7 is independent but should trail D1/D2 so the turn contract changes once. One schema-version bump covers the batch table, queue state, and provenance columns together.

## Rationale

The amendments proved these shapes by building against them: every interim the frontend ships today is a workaround for state the engine either already holds (mode record, changeset session id, terminal run vocabulary) or is the only party positioned to hold durably (queued turns, interrupt state, settlement). The chosen design closes each gap at the layer the a2a ADR already assigned durable truth to — the authoring store and its lifecycle feed — leaving the relay exactly the transient token/tool-call stream it was scoped to be. Reported settlement (O3) over inferred settlement (O2) follows from the adapter boundary: the engine maps runtime references, it does not watch a runtime; a command from the authenticated run owner is both honest and enforceable. Deleting the JOIN arm rather than layering a queue beside it follows the no-bridge doctrine and removes a silent-data-loss path in the same stroke. Deferring models (D6) is the same honesty the frontend already renders: a disabled pill with a reason beats a served list nothing honors.

## Consequences

- **Gains.** "Done" and "Failed" become wire truths; Stop stops a run instead of ending the conversation; queued prompts survive reload and race nothing; permission prompts recover after reload; the proposal card binds per-run; the autonomy control works pre-proposal; comment feedback becomes auditable data. Four shipped frontend interims (queue chip logic, staged-interrupt annex, session-actor-latest correlation, proposal-gated mode control) get their promised deletions.
- **Costs.** One authoring schema-version migration (batch table, provenance columns, queue state) — dev stores need the usual clean rebuild; two new lifecycle kinds (`run.settled`, `turn.queued`, plus `session.cancelled`) each cost an SSE-adapter case frontend-side; the settle command adds an owner-forgery surface that the owner-only authorization must actually enforce (a review focus, not a risk to accept).
- **Pitfalls.** The cancel-semantics cutover is behavior-visible: any consumer relying on cancel-collapses-session (the current frontend does — it bootstraps a fresh session after Stop) must cut over in the same release, not discover it; the queue promotion must stay inside the settle/cancel unit of work or a crash between settle and promote strands a queued turn; typed decision projection over historical opaque decisions must degrade per-record (`decision_unreadable`), never fail the list.
- **Pathways.** Settlement gives the a2a relay a durable anchor to reconcile against (truth recovery re-reads runs, never the relay); the feedback batch is the beachhead for the cross-repo continuation; run/turn provenance on changesets opens per-run audit rollups later without touching identity.

## Verification strategy

- **D1** — store tests: `Active → Completed/Failed` transition, terminal replay idempotency, owner-only authorization (a non-owner settle is a typed 403); SSE test proves `run.settled` reaches the feed once per settlement.
- **D2** — cancel leaves the session `Active`; session-cancel terminates both aggregates with both events; queue: cap enforced with typed 422, FIFO promotion inside the settle unit of work (crash-injection test: settle+promote is atomic), session cancel voids the queue with no promotion (queued turns readable but never runnable afterward), `joined` vocabulary gone from the codebase.
- **D3** — the route serves the existing store listing capped and truncation-marked in raise order with pending state explicit; typed decision round-trips the permission-decision write; a legacy opaque decision serves `decision_unreadable` without failing the page; a client that drops the `/execute` `awaiting_permission` response recovers the pending interrupt from the list (the subsumed advisory's recovery case, as a live test).
- **D4** — projection serves `session_id`/`run_id`/`turn_id` for a tool-dispatched proposal and `None` for a human one; pre-migration records still deserialize.
- **D5** — `GET /v1/mode` round-trips `POST /v1/mode` and serves the default record on a fresh store.
- **D7** — batch immutability under comment edits, cap and byte-bound enforcement, turn-reference fence violations typed; the turn record carries the batch id.
- **Full gate** — `just dev lint all` green (Rust fmt + clippy included) before any phase reports done; live-wire frontend suites re-run where adapters gain cases.
