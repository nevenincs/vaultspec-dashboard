# Figma Visual Parity

Produce hard visual evidence that a live UI matches or diverges from a Figma node.
Inspecting DOM, tests, screenshots in isolation, or MCP design context is **not**
sufficient — a valid run writes local image files and you must look at them.

## Hard gate — a run is complete only with all of these on disk

1. Figma reference PNG
2. Live implementation PNG (same pixel dimensions as the Figma node)
3. Split-comparison PNG
4. Direct alpha-overlay PNG
5. Pixel-difference PNG
6. JSON report with dimensions + pixel-delta stats
7. Static HTML review page (unless the user opts out)

Then **inspect the overlay and diff images** and report concrete deltas, not just
artifact paths.

## Workflow

1. **Identify inputs:** Figma `fileKey` + `nodeId` (both from the node's figma.com
   URL), the live URL of the implementation, a short `slug` for filenames (e.g.
   `hero`, `nav`), and the output directory (default `output/visual-compare/`,
   override with `--out`).
2. **Capture the Figma node** via the MCP screenshot tool and record the returned
   `width`/`height`. → `references/figma-mcp-capture.md`
3. **Download the Figma PNG** with the cross-platform helper:
   `node ${CLAUDE_SKILL_DIR}/scripts/fetch-figma-asset.mjs --url "<image_url>" --out output/visual-compare/<slug>-figma-<node-id-with-hyphen>.png`
4. **Capture the live route at the exact Figma dimensions** and generate the split,
   overlay, diff, and JSON report in one step (the HTML review page is generated by
   default; pass `--no-review` to skip it):
   `node ${CLAUDE_SKILL_DIR}/scripts/run-parity.mjs --slug <slug> --url <live-url> --figma output/visual-compare/<slug>-figma-<node-id-with-hyphen>.png --width <w> --height <h> [--wait-ms 6500] [--selector "<css>"] [--no-webgl] [--no-review] [--allow-resize] [--init-eval "<js>"] [--clip-selector "<css>"] [--vw 1440] [--vh 900]`
   - **Sub-components & transient states.** When the Figma node is a small
     sub-component (a control, a row) rather than a full screen, capturing the page
     at the tiny Figma viewport is meaningless. Use `--clip-selector "<css>"` to
     screenshot just that element from a real viewport (`--vw`/`--vh`, default
     1440×900), and add `--allow-resize` so the clipped element is scaled to the
     Figma dimensions for the diff. To compare a state that only appears after
     interaction (a zoomed scrubber, an open menu, a hovered row), pass
     `--init-eval "<js>"` — the JS runs in the page after the wait and before the
     screenshot (e.g. focus an element and dispatch key/pointer events to drive it
     into the state the Figma node depicts).
5. **Inspect** at least the overlay and diff PNGs. Read the report's `changedRatio`
   and `meanMaxChannelDelta` as evidence.
6. **Report concrete visual differences** — what diverges and where — not just paths.

The scripts are portable Node `.mjs` (Windows / macOS / Linux); they replace the
original PowerShell + `curl.exe` steps. They need `@playwright/test` with chromium
installed (auto-located, or set `FIGMA_PARITY_PLAYWRIGHT`).

## Failure modes to avoid

- **Incomplete tool discovery** — if no screenshot tool appears, search again:
  `Figma get_screenshot screenshot PNG selection node capture`.
- **Design context ≠ evidence** — an inline image from `get_design_context` does not
  satisfy the gate; write the PNGs.
- **Viewport mismatch** — capture live at the Figma dimensions, `deviceScaleFactor: 1`,
  `fullPage: false`. `compare-pngs.mjs` fails on a mismatch unless `--allow-resize`.
- **WebGL fallback** — for canvas/graph views, require `webglFallback: false` in the
  capture diagnostics.
- **Don't store the short-lived Figma asset URL** — keep only the PNG + metadata.
- **Don't add native image deps** — use browser Canvas via Playwright.
